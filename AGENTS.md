# 상세페이지 제작기 — 에이전트 가이드

이 저장소는 **개인 외주용 상세페이지 제작 도구**를 목표로 합니다.  
가장 강한 제품 제약은 **LLM·API 비용(토큰) 절감**이며, 구현·프롬프트·UX는 이를 깨지 않는 방향으로 맞춘다.

## 참고 문서

- `docs/feature-spec.md` — 기능 범위·우선순위·비기능
- `docs/cost-saving-policy.md` — 비용 정책 상세·체크리스트
- `prompts/section-copy.md` — 섹션 신규 생성
- `prompts/revise-section.md` — 섹션 수정·톤 정렬 (전체 문맥 없이)

---

## 1. 섹션 단위 생성 원칙

- **한 번의 모델 호출 = 한 섹션**이다. 한 요청에 “상세페이지 전체” 또는 여러 섹션 묶음을 생성하지 않는다.
- **컨텍스트는 슬롯화**한다. 프로젝트 전체 서사·원문 기획서를 붙이지 않고, 해당 섹션에 필요한 필드(상품명, USP, 스펙, 톤, 금지어 등)만 전달한다.
- **수정은 `revise` 경로**를 쓴다. 이미 존재하는 섹션 텍스트와 수정 지시만 넣고, 다른 섹션·전체 페이지를 다시 넣지 않는다 (`prompts/revise-section.md`).
- 새 섹션은 **`section-copy`** 한 세트만 사용하고, 결과는 앱 상태에 저장된 뒤 다음 섹션은 **별도 호출**로 진행한다.
- UI에서는 “미리보기 조립”은 **클라이언트 로컬에서** 하고, “한 번에 N섹션 자동 생성” 같은 배치는 **설계 단계에서 기본 제공하지 않는다** (비용 폭증·재시도 난비).

---

## 2. 전체 재생성 금지

- **금지**: 단일 사용자 액션 또는 단일 API로 **현재 페이지의 모든 섹션**을 통째로 다시 생성하는 동작 (“전체 다시 받기”, `regenerate-all`, “프로젝트 전체 카피 재생성” 등).
- **금지**: 공통 원인(예: 톤 변경)을 이유로 **전 섹션에 동일 긴 프롬프트 재전송**하는 일괄 파이프라인. 톤·용어 변경은 **섹션별로** `revise` 하거나, **짧은 전역 레퍼런스(몇 줄)** 만 바꿔 **해당 섹션만** 재호출한다.
- **허용**: 사용자가 명시적으로 연 **한 섹션**만 대상으로 한 재생성·수정 호출.
- **허용**: 내부 테스트용 스크립트가 섹션을 **순차** 호출하더라도, 각 호출은 **단일 섹션**이어야 하며 “전 페이지 한 방” 프롬프트는 두지 않는다.

코드 리뷰 시 “전체 루프로 매번 새 카피” 패턴이 보이면 **차단 대상**으로 본다.

## 3. 캐싱 전략

- **목표**: 동일 또는 의미적으로 동일한 입력에 대한 **중복 호출 제거**. 캐시는 정확도보다 비용 회피에 유리해야 하며, 민감 복붙 원문 장기 저장은 피한다.
- **캐시 키(권장 구성 요소)**  
  - 프롬프트·템플릿 버전(파일 해시 또는 버전 문자열, 예: `section-copy@v1`)  
  - 프로젝트 또는 상품 단위의 **정규화된 컨텍스트 요약**(해시; 원문 통째 저장 대신 필드별 정규화 후 해시)  
  - **섹션 ID·섹션 템플릿 타입**  
  - 모델 ID·temperature 등 **결과에 영향 주는 추론 파라미터**  
  - `revise`의 경우 **이전 결과 텍스트의 해시 + 수정 지시 문구 해시**(입력 변동 시 다른 키)
- **저장 계층**: 로컬/단일 사용자 전제에서는 **메모리 + (선택) SQLite/파일 KV** 수준으로 충분. 멀티 유저·서버 공유 시에만 외부 캐시를 검토한다.
- **TTL**: 짧은 TTL(예: 24~72h) 또는 프로젝트 단위 “캐시 비우기”로 **오래된 스타일·가격 문구** 잔존을 완화한다.
- **무효화**: 해당 섹션의 컨텍스트·수정 지시·프롬프트 버전이 바뀌면 **그 섹션 키만** 무효화. 전 프로젝트 컨텍스트가 바뀌면 **해당 프로젝트 접두 키**만 일괄 무효화 (가능하면; 아니면 TTL에 맡긴다).
- **적중 시**: 캐시에서 응답을 돌려주기 전, UI에 **“캐시됨” 표시(선택)** 로 사용자가 재호출해 신규 생성할지 판단할 수 있게 한다.
